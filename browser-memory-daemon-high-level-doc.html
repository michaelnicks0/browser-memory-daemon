1 What it does

The repo is a complete local browser-memory implementation, not a mockup: capture, policy, storage, search, UI, deletion, media sidecars, and real Chrome verification are all present.

🧠

Text-first recall

Visible Chrome page text becomes documents, visits, snapshots, chunks, and FTS rows in WSL. Text recall does not wait on media bytes.

🔎

Exact local search

SQLite FTS5 backs search, timeline, recent captures, document detail, snapshot text, and source snippets through the CLI, API, and local UI.

🛡️

Adjustable policy modes

`all`, `recall`, `balanced`, and `strict` are implemented. `all` removes built-in URL filtering/redaction but still honors explicit local block rules.

🎞️

Durable media sidecars

Browser IndexedDB queues, raw blob upload, X/Twitter CDP recovery, daemon public fetch, HLS handling, rolling cache gates, and bounded automatic spool recovery preserve media provenance without blocking text capture.

🖥️

Operator surfaces

A token-bootstrapped local dashboard, CLI, popup/options controls, HTTP API, doctor checks, policy rules, and deletion receipts cover day-to-day operation.

✅

Real browser verification

The test suite includes daemon pytest, extension node:test/build, and Chrome for Testing e2e for capture/search/forget/media/lifecycle behavior.

2 How it works

The runtime spine is simple: capture fast, classify locally, store text durably, fetch media lazily, and keep every operator read/delete path authenticated.

  1. 1
    Chrome observes the pageContent scripts extract visible text, media refs, scroll/dwell hints, and lifecycle signals while retaining hidden/form/editable/script/style/no-script skips.
  2. 2
    Service worker owns transportThe extension service worker attaches the bearer token, queues capture/media work, and posts only to the loopback WSL daemon.
  3. 3
    Policy classifies locallyThe daemon applies the selected policy mode plus persistent block rules; non-all modes redact before storage.
  4. 4
    Text is stored firstThe ingest path normalizes URLs, hashes text snapshots, writes complete cleaned text, provenance, chunks, and FTS rows to SQLite, then returns stable IDs.
  5. 5
    Media sidecars fill inBrowser and daemon workers fetch/upload blobs asynchronously under per-artifact, snapshot, domain, and global cache gates.
  6. 6
    Operator reads or forgetsCLI/API/UI surfaces expose exact search, recent/timeline/detail, doctor diagnostics, media cache controls, policy rules, and forget receipts.

3 The mental model

One mental model: Chrome is the sensor, WSL is the memory, and every retrieval/deletion path stays local and token-gated.

Chrome captures; WSL remembers. Media sidecars are asynchronous; exact text recall remains the source of truth.

6 The safety boundary

The product is powerful because it captures broadly by default. The safety story is locality, explicit operator control, and no hidden cloud path.

Runtime DB/state belongs under WSL XDG paths; blobs can live under a configured WSL-visible NAS root. The repo contains source, docs, generated visuals, and tests — not browser memory, tokens, cookies, or raw captures.
🛡️
The default all mode is an operator-selected recall posture: no daemon redaction and no built-in URL filtering. Explicit local block rules still apply. See security-model for the full contract.
